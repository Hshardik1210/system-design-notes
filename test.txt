If failure again:

attempt 2 → FAILED
attempt 3 → FAILED

After max retries:

PENDING	Do nothing or ensure queued
RETRY_PENDING	Let retry worker handle it
FAILED	Maybe retry manually or ignore
PROCESSING	Do nothing

1. What exactly is Kafka lag?

Kafka stores messages in partitions. Each consumer group tracks how far it has consumed using an offset.

Latest Kafka offset = 100000
Consumer offset     = 95000

Lag = 5000 messages

Meaning:

5000 messages are waiting to be processed
2. Why does lag happen?

Common reasons:

1. Too few workers
2. Workers are slow
3. Provider is slow, like SMS/email provider
4. DB updates are slow
5. Kafka topic has too few partitions
6. One partition is too hot
7. Worker errors causing retries
8. Large campaign/fan-out spike

Example:

Flash sale notification → 10 million users
Kafka receives huge traffic
Workers cannot process fast enough
Lag increases
3. Is Kafka lag always bad?

Not always.

Small temporary lag is normal.

Example:

Lag spikes for 2 minutes during traffic burst
Then workers catch up

That is okay.

Bad lag is:

Lag continuously increasing
Lag not reducing
Critical notifications delayed
4. What happens to users?

For notifications:

OTP / payment alert → dangerous if delayed
Marketing notification → okay if delayed
Order confirmation → should be reasonably fast

So you should not treat all notifications equally.

1. Scale consumers/workers

If workers are slow because there are too few of them:

Increase worker instances

Example:

Email workers: 5 → 20
Push workers: 10 → 100

But there is a limit:

In one consumer group, maximum parallelism is limited by number of Kafka partitions.

If topic has 10 partitions, only 10 consumers can actively consume in parallel in the same group.

2. Increase partitions

If you need more parallelism:

Increase topic partitions

Example:

notification-push topic:
10 partitions → 100 partitions

Then more workers can consume in parallel.

1. Where DB bottleneck can happen

In our flow:

Notification API
  → insert notification row

Workers
  → update status SENT / RETRY_PENDING / FAILED
  → insert attempt rows
  → read retry_pending rows

So DB is used for:

1. INSERT into notifications
2. INSERT into notification_attempts
3. UPDATE notification status
4. SELECT pending/retry records
5. Query notification history

If traffic is high, all of these hit the same DB.

2. Main DB bottlenecks
Bottleneck 1: Too many writes

Example:

1 million notifications/minute

For each notification, you may do:

1 insert into notifications
1 insert into notification_attempts
1 update notification status

That becomes:

3 million DB writes/minute

This can overload DB CPU, disk I/O, WAL/binlog, locks, and indexes.

Bottleneck 2: Index maintenance

You created:

CREATE UNIQUE INDEX uniq_notification_key
ON notifications(notification_key);

This is useful, but every insert must also update the index.

So insert is not only:

write row

It is also:

write row + update unique index + maybe other indexes

Too many indexes slow down writes.

Bottleneck 3: Hot rows

Suppose multiple workers update same notification row:

UPDATE notifications
SET status = 'RETRY_PENDING'
WHERE notification_id = 123;

If retries/duplicates happen, many processes may touch same row.

This causes row-level lock contention.

Bottleneck 4: Retry polling query

If retry worker keeps running:

SELECT *
FROM notifications
WHERE status = 'RETRY_PENDING'
AND next_retry_at <= now();

This can become slow if not indexed properly.

You need an index like:

CREATE INDEX idx_notifications_retry
ON notifications(status, next_retry_at);

Otherwise DB may scan many rows.

Bottleneck 5: Large notification history table

Notifications table grows very fast.

Example:

10 million notifications/day

After 30 days:

300 million rows

Queries, indexes, backups, and maintenance become heavier.

Solution:

partition by created_at
archive old data
move old history to cold storage
Bottleneck 6: Connection overload

If many workers run:

100 workers × 10 DB connections = 1000 connections

DB may hit connection limit.

So even before query load, DB can suffer from too many open connections.

Use proper connection pool limits.

Bottleneck 7: Slow queries from dashboards/history APIs

Example:

SELECT *
FROM notifications
WHERE user_id = 123
ORDER BY created_at DESC;

Without proper index:

CREATE INDEX idx_notifications_user_created
ON notifications(user_id, created_at DESC);

This can become slow and affect worker updates.

3. How to reduce DB bottleneck
1. Keep queue message small

Queue should carry:

{
  "notificationId": 123,
  "channel": "PUSH"
}

Worker fetches only what it needs.

2. Batch DB writes

Instead of updating one by one:

update notification 1
update notification 2
update notification 3

Use batch updates where possible.

3. Use proper indexes

Important indexes:

-- deduplication
CREATE UNIQUE INDEX uniq_notification_key
ON notifications(notification_key);

-- retry worker
CREATE INDEX idx_notifications_retry
ON notifications(status, next_retry_at);

-- user notification history
CREATE INDEX idx_notifications_user_created
ON notifications(user_id, created_at DESC);

-- attempts lookup
CREATE INDEX idx_attempts_notification
ON notification_attempts(notification_id);
4. Partition old data

For high volume:

notifications_2026_07
notifications_2026_08
notifications_2026_09

Usually partition by:

created_at

because notification history is time-based.

5. Separate operational DB from analytics

Do not run heavy analytics queries on the main notification DB.

Bad:

Production DB → delivery + reports + analytics

Better:

Production DB → delivery/status
Data warehouse → analytics/reporting
6. Use Kafka retry topics instead of heavy DB polling

Instead of constantly scanning DB for retries:

status = RETRY_PENDING

You can publish failed messages to delayed retry queues/topics:

notification-retry-1m
notification-retry-5m
notification-retry-30m
notification-dlq

DB still stores status, but Kafka drives retry timing.

Scaling workers helps Kafka lag, but can hurt DB.

10 workers → DB okay

1. Reduce DB writes

A bad design may do too many writes per notification:

1. Insert notification row
2. Insert attempt row
3. Update status to PROCESSING
4. Update status to SENT
5. Update provider response

For 1 million notifications, this becomes millions of DB writes.

A better design is to avoid unnecessary intermediate status updates.

Instead of:

PENDING → PROCESSING → SENT

You can often do:

PENDING → SENT

Only write PROCESSING if you really need it.

2. Batch DB writes

Instead of each worker updating DB one row at a time:

UPDATE notifications SET status = 'SENT' WHERE notification_id = 1;
UPDATE notifications SET status = 'SENT' WHERE notification_id = 2;
UPDATE notifications SET status = 'SENT' WHERE notification_id = 3;

Batch it:

UPDATE notifications
SET status = 'SENT', sent_at = now()
WHERE notification_id IN (1, 2, 3, 4, 5);

Same for inserting attempts:

Insert 100 attempts together
instead of 100 separate inserts

This reduces DB round trips.

3. Add the right indexes

For retry processing:

CREATE INDEX idx_notifications_retry
ON notifications(status, next_retry_at);

For user notification history:

CREATE INDEX idx_notifications_user_created
ON notifications(user_id, created_at DESC);

For deduplication:

CREATE UNIQUE INDEX uniq_notification_key
ON notifications(notification_key);

But don’t add too many indexes, because every index slows down inserts and updates.

4. Partition the notifications table

Notification tables grow very fast.

Example:

10 million notifications/day
300 million/month

So partition by time:

notifications_2026_07
notifications_2026_08
notifications_2026_09

Why this helps:

Recent notifications → hot partition
Old notifications → archived or cold partition

User history queries and cleanup become easier.

5. Archive old data

Do not keep all notification history in the main operational DB forever.

Example policy:

Last 30 days → main DB
Older than 30 days → S3 / data warehouse / archive DB

This keeps your main DB small and fast.

6. Use Kafka retry topics instead of DB polling

Bad retry design:

SELECT *
FROM notifications
WHERE status = 'RETRY_PENDING'
AND next_retry_at <= now();

If this runs constantly on huge table, DB load increases.

Better:

notification-main
notification-retry-1m
notification-retry-5m
notification-retry-30m
notification-dlq

Flow:

Worker fails sending
→ update attempt/status
→ publish message to retry topic
→ retry later

Now Kafka controls retry timing, not constant DB scanning.

7. Control worker concurrency

If Kafka lag happens, people often add more workers.

But too many workers can overload DB.

Example:

10 workers → DB okay
100 workers → DB overloaded

So limit worker concurrency:

max 50 workers for push
max 10 workers for email
max 5 workers for SMS

Also use connection pool limits:

Each worker should not open too many DB connections
8. Separate high-priority and low-priority notifications

Don’t let marketing traffic overload the same DB path as OTP/security alerts.

Use separate queues/topics:

notification-critical
notification-normal
notification-marketing

And separate workers.

For example:

OTP/security → strict tracking in DB
Marketing → lighter tracking / batch tracking

For marketing, you may not need to store every single attempt in detail.

9. Use caching where possible

Some data is repeatedly read by workers:

user preferences
templates
provider config
rate limits
device tokens

Cache these in Redis or in-memory cache.

Example:

template_id → template body
user_id → notification preferences

This reduces DB reads.

10. Split read and write workload

Operational writes:

insert notification
update status
insert attempts

User-facing reads:

show notification history

Analytics reads:

delivery report

Don’t run all on same DB.

Better:

Instead of scanning attempts table every time:

Maintain summary counters:

notification_metrics_daily

date | channel | sent_count | failed_count

Workers update metrics asynchronously.

If one DB cannot handle writes even after optimization, then shard.

user_id

    status VARCHAR(50),
    created_at TIMESTAMP,
    sent_at TIMESTAMP
);

And add FK if same DB:

ALTER TABLE notifications
ADD CONSTRAINT fk_notifications_template
FOREIGN KEY (template_id)
REFERENCES notification_templates(template_id);

Important: event_type remains same, template_id changes.

event_type = ORDER_CONFIRMED

Old template:
template_id = uuid-1
is_active = false

New template:
template_id = uuid-2
is_active = true

For selecting current template:

SELECT *
FROM notification_templates
WHERE event_type = 'ORDER_CONFIRMED'
  AND channel = 'PUSH'
  AND language = 'en'
  AND is_active = true;

You should also enforce only one active template per event/channel/language.

In PostgreSQL:

CREATE UNIQUE INDEX uniq_active_template
ON notification_templates(event_type, channel, language)
WHERE is_active = true;

Final model:

event_type = why notification is sent
template_id UUID = exact template row used
is_active = use this template for new notifications or not

So yes — UUID as primary key is perfectly fine.
